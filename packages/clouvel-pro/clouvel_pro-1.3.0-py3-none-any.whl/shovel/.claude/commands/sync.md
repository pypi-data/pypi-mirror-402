# /sync - 프로젝트 동기화 및 정리

> **목적**: docs 전체를 분석해서 현황 파악 → 우선순위 재계산 → 실행 계획 업데이트
> **사용 시기**: 프로젝트 재개 시, 문서가 복잡해졌을 때, 방향성 재정립 필요 시

---

## 실행 단계

### 1단계: 전체 문서 스캔 (5분)

```bash
# docs/ 폴더 전체 읽기
ls docs/

# 각 문서 확인
- PRD.md          → 제품 스펙
- PLAN.md         → 실행 계획
- BACKLOG.md      → 대기 중 아이디어
- PROGRESS.md     → 진행 상황
- [기타 .md]      → 추가 문서들
```

**체크 항목:**
```markdown
□ 각 문서의 마지막 업데이트 시점
□ PRD와 PLAN의 일관성
□ 완료 표시된 항목들
□ 중복된 내용
□ 모순되는 계획
```

---

### 2단계: 현황 분석 리포트 생성

**자동으로 생성할 내용:**

```markdown
# 프로젝트 현황 분석

## 📊 문서 상태
- PRD: 최종 수정 [날짜]
  - 전체 기능: [N]개
  - 완료: [N]개 ([%]%)
  - 진행 중: [N]개
  - 대기: [N]개

- PLAN: 최종 수정 [날짜]
  - 계획된 단계: [N]개
  - 현재 단계: [Phase X]
  - 지연 여부: [Yes/No]

- BACKLOG: [N]개 아이템
  - 우선순위 높음: [N]
  - 우선순위 중간: [N]
  - 우선순위 낮음: [N]

## ⚠️ 발견된 문제
- 중복: [목록]
- 모순: [목록]
- 누락: [목록]
- 오래된 정보: [목록]

## ✅ 완료됐으나 미체크
[PRD에 완료 표시 안 된 것들]

## 🚧 진행 중 확인 필요
[PROGRESS.md vs 실제 코드 차이]
```

### 3단계: 우선순위 재계산

**RICE 스코어 기반 재정렬:**

```markdown
## 우선순위 재계산 결과

| 기능 | 기존 순위 | 새 순위 | RICE | 변경 이유 |
|------|----------|---------|------|-----------|
| F-001 | P0 | P0 | 850 | 유지 |
| F-003 | P1 | P0 | 920 | ↑ 고객 요청 증가 |
| F-005 | P0 | P2 | 120 | ↓ 기술 난이도 높음 |

## 변경 사항 요약
- P0 → P2: [목록] (왜?)
- P1 → P0: [목록] (왜?)
- 새로 추가: [목록] (BACKLOG에서)
```

---

### 4단계: 사용자 질문 (대화형)

**Claude가 물어볼 질문들:**

```
1. 완료 표시 확인
   "PRD F-001이 코드에 구현되어 있는데 완료 체크가 안 돼있어요.
    완료 처리할까요?"

2. 우선순위 변경 확인
   "F-003의 RICE 점수가 올라가서 P0으로 승격 추천합니다.
    동의하시나요?"

3. BACKLOG 승격
   "BACKLOG의 B-005가 고객 요청이 많아졌어요.
    PRD로 올릴까요?"

4. 막힌 부분 확인
   "PROGRESS.md에 F-002가 2주째 진행 중이에요.
    막힌 부분이 있나요?"

5. 새로운 방향성
   "최근 고객 피드백이나 비즈니스 방향 변경이 있었나요?"
```

**사용자 답변 예시:**
```
1. → "네, 완료 처리해주세요"
2. → "아니요, P1 유지. 기술 리스크 때문에"
3. → "네, P1으로 추가"
4. → "YouTube API 인증 부분에서 막혔어요"
5. → "수익화를 더 급하게 해야 해요"
```

### 5단계: 전체 로드맵 조정

**업데이트할 내용:**

```markdown
## 분기별 로드맵 재조정

### Q1 2026 (현재)
**기존 목표:**
- MVP 출시
- 베타 테스터 10명

**조정된 목표:**
- 핵심 기능만 MVP (F-001, F-003)
- 수익화 우선 (F-002는 나중으로)

### Q2 2026
**기존:**
- 전체 기능 완성

**조정:**
- 고객 피드백 반영
- F-005 추가 (고객 요청 많음)
```

### 6단계: 출력물 생성

**자동으로 생성/업데이트할 파일:**

#### A) PLAN.md (업데이트)

```markdown
# [프로젝트] 실행 계획 v2.0

> 최종 업데이트: [날짜]
> 다음 동기화: [추천 날짜]

## Phase 1: 핵심 MVP (2주)
- [x] F-001: 기능 1
- [ ] F-003: 기능 3 (우선순위 상승)
- [~] F-002: 기능 2 (진행 중, 막힘)

## Phase 2: 수익화 (3주)
[조정된 계획]

## 변경 이력
- 2026-01-14: F-003 P1→P0 (고객 요청)
- 2026-01-14: F-005 P0→P2 (기술 리스크)
```

#### B) TODO.md (신규 생성)

```markdown
# 이번 주 할 일 (Week of [날짜])

## 🔥 긴급 (오늘~내일)
- [ ] F-002 디버깅 (YouTube API 인증 해결)
- [ ] F-003 /plan 작성

## 🎯 중요 (이번 주)
- [ ] F-001 /verify 통과
- [ ] F-003 구현 시작
- [ ] PRD 완료 표시 업데이트

## 📦 여유 있으면
- [ ] BACKLOG 검토
- [ ] 기술 문서 정리

## 🚫 하지 않을 것
- F-005 (다음 분기로)
- B-001~B-010 (BACKLOG 유지)
```

#### C) IMPROVEMENTS.md (개선안)

```markdown
# 프로젝트 개선 제안

## 문서 관리
- [ ] PRD와 PLAN 동기화 자동화
- [ ] 완료 기준 명확화 (DoD 추가)

## 프로세스
- [ ] 주간 /sync 루틴 추가
- [ ] RICE 재계산 주기 정하기

## 기술 부채
- [ ] F-002 리팩토링 (구현 방식 재검토)
- [ ] 테스트 커버리지 향상

## 비즈니스
- [ ] 수익화 타임라인 구체화
- [ ] 고객 피드백 수집 시스템
```

## 사용 예시

### 시나리오 A: 2주 만에 프로젝트 재개

```bash
# PathcraftAI 폴더에서
/sync

→ Claude: "docs를 분석 중입니다..."
→ Claude: "F-002가 2주째 진행 중이네요. 막힌 부분이 있나요?"
→ 나: "YouTube API 인증에서 막혔어요"
→ Claude: "우선순위를 재계산했습니다. F-003을 먼저 하는 게 어때요?"
→ 나: "좋아요"
→ Claude: "PLAN.md, TODO.md 업데이트 완료했습니다."
```

### 시나리오 B: 문서 혼돈 상태

```bash
# AdGen 폴더에서
# (PRD 10번 수정, PLAN 5번 수정, BACKLOG 20개 추가)

/sync

→ Claude: "중복된 기능을 발견했어요: [목록]"
→ Claude: "PRD와 PLAN이 안 맞아요: [차이점]"
→ Claude: "BACKLOG에서 3개를 PRD로 올리는 게 좋겠어요"
→ 나: [답변]
→ Claude: "전체 정리 완료. IMPROVEMENTS.md에 개선안 추가했어요."
```

## 주의사항

### ❌ 하지 않는 것

```
- 코드 수정 (분석만, 구현 안 함)
- 자동 커밋 (사용자 확인 후)
- PRD 내용 변경 (재정렬만)
```

### ✅ 반드시 하는 것

```
- 모든 변경사항 사용자 확인
- 변경 이력 기록
- 백업 권장 (git commit 먼저)
```

## 다음 단계

```
1. 업데이트된 PLAN.md 확인
2. TODO.md의 긴급 항목부터 시작
3. /plan [다음 기능] 실행
```

```
- 방향성 변경: /sync (전체 재정렬)
- 문서 10개+ 수정: /sync (정리)
```

```markdown
□ docs/ 전체 파일 읽음
□ PRD 완료율 계산

□ 중복/모순 찾음
□ RICE 재계산
□ 사용자 질문 (4-5개)
□ PLAN.md 업데이트
□ TODO.md 생성
□ IMPROVEMENTS.md 생성
□ 변경 이력 기록
□ 다음 액션 제시
```
